## Context

当前“面试 Agent”已经具备分类、本地检索、结构化回答、追问和会话历史等能力，但整体交互仍然是响应式的：用户提出一个问题，系统围绕这一次提问做分类、检索和回答。这个模式对“快速查一道题”是有效的，但对“连续准备某家公司某岗位面试”这种训练型目标并不够。

求职准备场景下，系统真正需要承担的是“训练组织者”的角色，而不只是“知识回答者”。这意味着后端不能只围绕若干独立接口建模，而需要开始围绕训练状态建模：当前目标是什么、当前练哪个专题、这题答得怎么样、下一步该追问还是切题、什么时候该做小结。

这次设计聚焦于把现有面试 Agent 收敛成一个最小可行的教练模式 MVP。重点不是做一个全自治的复杂 Agent，而是先建立一个半主动、可控、可解释、可复盘的训练闭环。

## Goals / Non-Goals

**Goals:**
- 让面试 Agent 支持“训练模式”，由系统围绕用户目标主动出题和推进练习。
- 建立一个足够小但完整的训练状态机，用于驱动目标设置、专题选择、出题、作答、评估和下一步决策。
- 为训练过程定义稳定的结构化对象模型，包括目标、训练会话、答题尝试、评估结果和阶段总结。
- 明确 `EVALUATE` 与 `DECIDE_NEXT` 的职责边界，避免让大模型同时承担流程控制和内容评估。
- 让第一版就能支持“主动出题 -> 用户回答 -> 评估 -> 追问/切题 -> 小结”的最小闭环。

**Non-Goals:**
- 不在第一版中实现复杂的长期学习计划、日程编排或跨天任务系统。
- 不在第一版中构建高度细粒度的用户能力画像或精确打分体系。
- 不让大模型完全自主决定所有状态跳转，保持规则驱动的节奏控制。
- 不把训练模式和开放式问答模式混成一个不可预测的单一交互流。
- 不在这次设计中追求完全通用的自治 Agent，仍然限定在“面试训练”这个垂直任务域。

## Decisions

### 1. 采用“问答模式 / 训练模式”双模式，而不是直接替换现有 Agent

面试 Agent 将保留当前问答式能力，并新增一条“训练模式 / 教练模式”路径。训练模式用于围绕目标主动推进；问答模式仍保留给用户快速查题或自由提问。

选择原因：
- 训练型交互和开放式问答的心智不同，混在一起容易导致系统行为飘忽不定。
- 当前系统已有稳定的问答能力，直接替换会放大回归风险。
- 双模式可以让训练闭环逐步演进，而不是一上来就要求所有 Agent 行为统一。

考虑过的替代方案：
- 直接把现有面试 Agent 全部改造成训练模式：风险高，用户也可能失去快速问答入口。
- 只保留问答模式，不引入独立训练模式：难以形成“系统主动推进”的教练感。

### 2. 用最小状态机驱动训练闭环

第一版训练状态机固定为：

```text
GOAL_SET
  -> TOPIC_PICK
  -> ASK
  -> WAIT_ANSWER
  -> EVALUATE
  -> DECIDE_NEXT
```

其中 `DECIDE_NEXT` 只允许四种动作：
- `FOLLOW_UP`
- `NEXT_QUESTION`
- `SWITCH_TOPIC`
- `SUMMARY`

选择原因：
- 这个状态数量足够小，容易验证，也足够支撑“教练模式”的产品差异。
- 状态机比完全自由的 LLM 决策更稳定、更容易调试。
- 把动作数量收敛到四种，可以把后端复杂度控制在 MVP 范围内。

考虑过的替代方案：
- 更复杂的多阶段状态机：会过早放大系统复杂度。
- 完全无状态的“每轮单独回答”：无法体现训练推进和节奏控制。

### 3. 目标建模保持轻量，不先做长期学习计划

第一版 `Goal` 只承载训练闭环必须的字段，例如：
- target company
- target role
- training mode
- topic preference

不在第一版中引入复杂的 deadline、日程计划、学习阶段树和多日路线编排。

选择原因：
- 教练模式真正的最小门槛是“围绕目标推进”，不是“做一个完整学习管理系统”。
- 轻量目标对象已足够改变系统行为，让 Agent 从“回答问题”变成“帮助准备”。
- 目标过重会把 MVP 推向长期任务系统，而不是训练闭环系统。

考虑过的替代方案：
- 直接上完整 `StudyPlan`：价值高，但容易在第一版做得太重。
- 完全不做目标对象：系统很难体现专题优先级和训练模式差异。

### 4. 题目来源优先复用本地知识资产，不追求完全自由生成

训练模式的主动出题优先使用三类来源：
- 本地帖子高频题
- 归并后的专题题库代表题
- 知识点模板改写题

选择原因：
- 这些来源都能利用当前系统已经沉淀的数据资产。
- 与直接让 LLM 自由生成相比，更容易保持和样本库一致。
- 更有利于“来源可追溯”和“贴近真实面经”的产品价值。

考虑过的替代方案：
- 完全依赖模型自由出题：灵活但不稳，也不容易解释题目从何而来。
- 完全固定题库：稳定，但训练多样性不足。

### 5. `EVALUATE` 负责输出训练信号，不直接输出最终动作

`EVALUATE` 的职责是把用户回答转成结构化训练信号，而不是直接决定流程跳转。第一版推荐输出：
- `answerQuality`
- `keyPointCoverage`
- `factualRisk`
- `expressionClarity`
- `shouldFollowUp`
- `weaknessTags`
- `confidence`

以及给用户展示的反馈：
- `strengths`
- `gaps`
- `suggestion`

选择原因：
- 后续动作不仅取决于当前回答，还取决于训练模式、追问次数、当前专题进度和当前轮次长度。
- 如果让 `EVALUATE` 直接给出最终动作，会把“局部判断”和“全局节奏控制”混在一起。
- 结构化输出更适合后续累计薄弱点、专题表现和训练复盘。

考虑过的替代方案：
- 让 LLM 直接输出 `nextAction`：短期省事，但后续很难做稳定控制。
- 只返回自然语言评语：人可读，但机器很难稳定消费。

### 6. `DECIDE_NEXT` 采用规则驱动，先保证节奏稳定

`DECIDE_NEXT` 消费以下信号：
- 当前回答质量
- 当前题追问次数
- 当前专题累计表现
- 当前轮训练长度
- 当前训练模式
- 当前专题优先级

第一版的核心原则是：
- 回答差：优先追问，但限制追问次数
- 回答一般：速刷模式倾向下一题，深挖模式允许追问
- 回答好：一般推进下一题或切专题
- 当前轮达到阈值：优先生成小结

选择原因：
- 训练系统最重要的是节奏稳定，而不是表面上的“全自动聪明”。
- 规则层更容易解释和调优，也便于通过日志复盘决策路径。
- 后续如果积累足够数据，可以再逐步放开更复杂的策略层。

考虑过的替代方案：
- 完全交由模型决定追问还是切题：行为不可预测，用户体验容易不稳定。
- 完全线性固定流程：太像刷题器，缺少教练判断。

### 7. 训练过程先围绕四个核心对象建模

第一版训练模式围绕以下对象设计：

#### Goal
表示“准备什么”
- company
- role
- mode
- topic preference

#### TopicProgress
表示“当前专题练到哪”
- current topic
- answered count
- recent quality trend
- follow-up count

#### Attempt
表示“一次答题尝试”
- question
- source evidence
- user answer
- evaluation result
- next action

#### ReviewSummary
表示“一轮训练的小结”
- topics covered
- strengths
- weaknesses
- recommended next focus

选择原因：
- 这四个对象已经足够支撑 MVP 闭环。
- 比起一开始建过多表和过多模型，这种对象最贴近训练流程本身。
- 后续扩展 `StudyPlan`、`WeaknessProfile`、长期画像时，也容易自然演化。

## Risks / Trade-offs

- [训练模式边界不清，可能和现有问答模式互相干扰]  
  -> 通过明确双模式入口和状态机起点隔离交互路径。

- [`EVALUATE` 输出不稳定会导致后续节奏乱]  
  -> 第一版优先用固定结构字段约束输出，而不是开放式评语。

- [决策规则过度复杂会让 MVP 落地困难]  
  -> 保持四种动作和少量关键输入，优先保证规则可解释。

- [如果过度依赖规则，训练显得机械]  
  -> 在“内容生成”上继续使用 LLM，在“流程推进”上使用规则，保持平衡。

- [训练过程新增状态和记录后，持久化模型会变复杂]  
  -> 第一版只保存训练闭环必须的数据，不急于引入长期学习管理能力。

## Migration Plan

1. 明确训练模式的入口和与现有问答模式的关系。
2. 为训练模式定义最小状态机和核心对象。
3. 为 `EVALUATE` 定义稳定的结构化输出格式。
4. 为 `DECIDE_NEXT` 确定少量、可解释的规则。
5. 设计训练会话、答题尝试和阶段总结的持久化边界。
6. 最后再扩展前端交互，承载主动出题、作答输入、反馈和小结展示。

回滚策略：
- 如果训练模式复杂度或体验不可控，可以保留问答模式不受影响，并将训练模式隐藏在实验入口之后。
- 如果评估或决策质量不稳定，可以先退回“主动出题 + 简单反馈”的弱闭环模式，而不是完全移除训练入口。

## Open Questions

- 第一版训练模式是否只支持“高频速刷 / 深度追问”两种模式，还是需要更多训练模式？
- `TopicProgress` 是否需要单独持久化，还是暂时依附在训练会话中即可？
- `ReviewSummary` 是每 3 题、每 5 题还是专题切换时生成，哪个节奏最合适？
- 第一版是否允许用户在训练过程中手动要求“跳题 / 切专题 / 直接总结”？
- 后续如果引入更长期的薄弱点画像，应该在本次 change 中先预留字段，还是等训练模式跑通后再扩展？
