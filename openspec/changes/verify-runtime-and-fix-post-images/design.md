## Context

项目当前已经迁移到 React + Vite，并通过 `local_app_server.py` 提供本地 API 与静态托管。帖子图片主要来自小红书搜索结果中的 `coverUrl`，展示位置包括采集候选卡片、帖子库卡片和帖子详情弹层。

图片加载不出来可能有多种原因：搜索结果没有返回图片 URL、图片 URL 已过期、远端图片限制外链、浏览器 mixed content/CORS/referrer 策略导致失败，或 React 组件没有处理 `img.onerror`。这类问题即使不影响数据入库，也会显著降低帖子库浏览体验，因此需要在功能巡检中一起修复。

## Goals / Non-Goals

**Goals:**

- 建立可重复执行的本地功能巡检清单，覆盖构建、后端测试、启动和核心页面访问。
- 排查帖子图片加载失败的主要原因，记录是数据问题、前端展示问题还是远端访问限制。
- 为采集候选卡片、帖子库卡片和详情弹层增加稳定的图片失败降级体验。
- 如确认为远端外链限制，评估并实现可选的本地图片代理或图片可用性检测。
- 保证修复后不破坏帖子库布局，不影响无图片帖子阅读。

**Non-Goals:**

- 不改变小红书搜索 API 的核心调用方式。
- 不修改 SQLite schema。
- 不缓存或批量下载第三方图片到本地数据库。
- 不承诺所有第三方图片都能永久显示；目标是失败时仍有优雅降级。

## Decisions

1. 先前端降级，再考虑后端代理。

   前端 `img.onerror` 和占位卡片是最低风险方案，即使图片 URL 无效或远端拒绝外链，也能保持页面可读。后端图片代理会增加网络、缓存、安全和版权边界，只有在确认大量图片因 referrer/外链限制失败时才启用。

2. 图片状态由组件内部管理，不写入帖子数据。

   图片是否加载成功是运行时状态，不应污染 SQLite 中的帖子记录。卡片组件可以根据 `coverUrl` 和加载结果显示真实图片或占位封面。

3. 巡检脚本以 smoke check 为主。

   先覆盖 `npm run build`、`python -m unittest test_local_app_server.py`、`/api/health`、首页 HTML 标记和关键路由渲染。涉及真实小红书搜索需要 Cookie，不适合作为自动化必跑项，可作为手动巡检步骤。

4. 图片代理接口保持可选。

   如果需要新增代理，建议使用 `GET /api/image-proxy?url=...`，只允许 `http/https` URL，并设置合理超时、大小限制和错误响应，避免变成开放代理。

## Risks / Trade-offs

- [Risk] 图片外链本身过期或被拒绝访问 → Mitigation：前端占位和错误提示兜底，必要时提供可选代理。
- [Risk] 后端图片代理被滥用 → Mitigation：限制协议、超时、响应大小和内容类型，不代理非图片内容。
- [Risk] 自动化巡检依赖真实 Cookie → Mitigation：自动化只覆盖无凭据路径，真实搜索作为手动检查项。
- [Risk] 占位图过多降低信息密度 → Mitigation：占位卡片展示公司、标题、日期和类型，尽量保留浏览价值。
- [Risk] 图片失败状态导致 React 组件重复渲染 → Mitigation：按图片 URL 维护轻量本地状态，URL 变化时重置状态。
