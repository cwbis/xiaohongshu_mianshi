## ADDED Requirements

### Requirement: 自动识别面试问题知识领域
系统 SHALL 根据用户输入的问题识别主要知识领域、相关标签和问题意图。

#### Scenario: 识别明确技术问题
- **WHEN** 用户提问“Redis 扣减库存如何保证缓存和数据库一致？”
- **THEN** 系统返回包含“缓存一致性”“Redis”“高并发库存”等标签的分类结果

#### Scenario: 识别模糊追问
- **WHEN** 用户在已有会话中追问“为什么不用先删缓存？”
- **THEN** 系统结合当前会话上下文判断该追问属于原问题的缓存一致性子问题

### Requirement: 检索相关帖子和知识上下文
系统 SHALL 基于问题、领域标签、公司和岗位过滤条件，从本地帖子库、题目库和知识点库检索相关上下文。

#### Scenario: 从帖子库检索上下文
- **WHEN** 用户提出一个面试问题且帖子库存在相关内容
- **THEN** 系统返回若干相关帖子片段，并包含来源标题、来源类型和匹配原因

#### Scenario: 上下文不足
- **WHEN** 本地数据中没有足够相关内容
- **THEN** 系统仍可生成通用回答，但必须在结果中标记本地上下文不足

### Requirement: 生成面试级结构化回答
系统 SHALL 结合检索上下文生成适合真实面试表达的总-分结构回答。

#### Scenario: 生成标准回答
- **WHEN** 用户提交问题并存在可用大模型配置
- **THEN** 系统返回一句话总结、分点回答、项目落地示例、易错点和可能追问

#### Scenario: 大模型配置缺失
- **WHEN** 用户未配置可用 API Key
- **THEN** 系统拒绝发起大模型请求，并返回可读的配置提示

### Requirement: 支持追问会话
系统 SHALL 保存 Agent 会话上下文，并在用户追问时复用原问题、上一轮回答、领域标签和检索来源。

#### Scenario: 基于上一轮回答继续追问
- **WHEN** 用户在回答后继续提问“为什么用滑动窗口限流？”
- **THEN** 系统将该问题作为当前会话追问处理，并返回聚焦该追问的补充回答

#### Scenario: 会话历史可恢复
- **WHEN** 用户重新打开应用并查看历史 Agent 会话
- **THEN** 系统从本地 SQLite 恢复问题、回答、追问和来源上下文

### Requirement: 回答来源可追溯
系统 SHALL 在 Agent 回答中展示用于生成回答的本地来源。

#### Scenario: 回答包含来源
- **WHEN** 系统使用本地帖子或知识点作为上下文
- **THEN** 回答结果包含来源列表，用户可以查看来源标题、类型和摘要
