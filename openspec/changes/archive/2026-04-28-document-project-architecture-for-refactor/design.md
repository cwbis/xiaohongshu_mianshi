## Context

OfferScope 当前是一个原生前端加 Python 本地服务的本地工作台。前端由 `index.html` 加载多个全局脚本，`app.js` 负责大部分状态、页面渲染、事件绑定和业务流程；后端由 `local_app_server.py` 同时承担静态文件服务、SQLite 存储 API、小红书 API 代理和服务启动控制；浏览器扩展仍保留桥接能力；OpenSpec 中已有本地持久化和本地运行时相关规范。

项目正在快速迭代，近期已经加入 SQLite、桌面启动器、帖子库分组浏览、小红书风格详情页、搜索分页和刷新等功能。后续重构前需要一份完整说明文档，帮助判断哪些模块可以拆、哪些行为必须保留、哪些历史代码可以清理。

## Goals / Non-Goals

**Goals:**

- 输出一份 `docs/project-refactor-guide.md`，作为后续重构的共同上下文。
- 梳理项目的业务目标、页面结构、模块职责、数据流、接口、持久化、启动方式、扩展桥接和测试方式。
- 标记当前重构风险点和建议拆分顺序，尤其是 `app.js` 职责集中、前端全局状态膨胀、旧扩展桥接代码残留、文档编码不一致等问题。
- 让后续重构可以按阶段推进，每个阶段都能对应到明确文件和行为边界。

**Non-Goals:**

- 本次不重构业务代码，不拆分 `app.js`，不调整 API 或数据库 schema。
- 本次不清理历史功能代码，只记录保留、待验证、可删除的判断依据。
- 本次不替代 README；README 继续保留快速启动和基础使用说明。
- 本次不归档其他 OpenSpec change。

## Decisions

### Decision: 新增独立 docs 文档，而不是继续扩写 README

README 当前承担快速启动、数据存储和基础功能说明。后续重构需要更细的模块边界、依赖关系和风险分析，如果继续写进 README，会让启动说明变重。

备选方案是更新 `技术文档.md` 或 `页面重构技术文档.md`。这些文档更像历史阶段记录，存在内容过期和口径不一致的问题；新增 `docs/project-refactor-guide.md` 更容易作为后续重构入口。

### Decision: 文档按“现状梳理 + 重构建议”组织

文档应先描述当前项目真实运行方式，再给建议。这样后续重构者可以先理解现有系统，再判断是否采纳建议，避免文档变成只有目标架构、缺少现状依据的愿望清单。

### Decision: 保留文件级索引和数据流说明

当前项目没有构建系统和模块导入，很多能力依赖脚本加载顺序和全局对象。文档必须列出核心文件职责、全局对象关系、前后端 API、数据库表和页面事件链路，否则后续拆分时容易漏掉隐式依赖。

### Decision: 用风险等级标注重构优先级

重构建议按高、中、低风险标注。高风险项包括 `app.js` 拆分、存储接口变更、小红书运行时代理；中风险项包括搜索页体验重构、帖子库视图拆分；低风险项包括文档整理、乱码修复、死代码清理。

## Risks / Trade-offs

- [Risk] 文档很快过期 → [Mitigation] 文档中记录最后梳理日期，并把“需要同步更新的触发条件”写清楚。
- [Risk] 只写建议而缺少现状依据 → [Mitigation] 每个重构建议都关联当前文件、模块或行为。
- [Risk] 误把历史代码判断为无用代码 → [Mitigation] 只在文档中标记“疑似可清理”，实现阶段再通过搜索、测试和人工确认删除。
- [Risk] OpenSpec 主规范中仍有历史乱码 → [Mitigation] 在说明文档中列为文档债务，不在本次变更内修复。
- [Risk] 文档过长导致不好用 → [Mitigation] 开头提供“重构快速索引”，正文再展开细节。
