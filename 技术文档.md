# 小红书面经分析应用技术文档

## 1. 文档目标

本文档描述当前版本的技术实现方式、核心模块、数据流、插件采集方案和自动填充桥接方案，便于继续开发。

## 2. 当前技术方案概览

当前版本采用“静态前端分析台 + Chrome 插件采集原型”的方式实现，不依赖后端服务，不依赖数据库，也不依赖打包构建工具。

技术选型如下：

- `HTML`：页面结构
- `CSS`：页面样式与响应式布局
- `JavaScript`：帖子库管理、分析逻辑、报告生成
- `Chrome Extension Manifest V3`：当前帖子页采集
- `window.postMessage + chrome.runtime messaging`：分析台与扩展之间的自动填充桥接

```text
.

```

各文件职责如下：

- `index.html`：页面结构，定义配置区、录入区、帖子库和分析区
- `styles.css`：页面主题、布局、组件样式、响应式规则
- `analysis-core.js`：可复用的分析核心，供前端和 Node 测试共用
- `app.js`：帖子库管理、本地存储、导入导出、自动填充和页面交互
- `extension/background.js`：统一处理自动填充请求，选择标签页并调度采集
- `extension/page-bridge.js`：运行在分析台页面上的桥接脚本，负责页面与扩展之间的消息转发
- `extension/content.js`：运行在小红书页面上的采集脚本
- `extension/popup.*`：手工采集插件界面
- `test-analysis.js`：分析核心的 smoke test

## 4. 前端页面结构

页面主要分为两个区域。

### 4.1 配置与录入区

包含：

- 公司输入框
- 岗位输入框
- 搜索词输入框
- 单条帖子标题、作者、发布时间、来源链接输入框
- 单条帖子正文输入框
- 保存到帖子库按钮
- 示例填充按钮
- JSON 导入按钮
- JSON 导出按钮
- 根据当前页面自动填充按钮
- 根据链接自动填充按钮
- 批量链接导入输入框
- 批量导入开始/停止按钮
- 批量任务状态区

### 4.2 帖子库与分析区

包含：

- 样本量概览
- 帖子库列表
- 高频关键词标签
- 常见问题列表
- 流程观察列表
- Markdown 报告输出区

## 5. 数据流

当前系统的执行流程如下：

1. 用户输入公司、岗位、搜索词
2. 用户通过手工录入、自动填充或导入 JSON 增加帖子
3. 系统将帖子统一转换为结构化记录
4. 帖子库保存到浏览器 `localStorage`
5. 分析核心对帖子库执行关键词统计、问题提取和流程提取
6. 系统生成 Markdown 报告并渲染到页面
7. 用户可复制报告，或导出帖子库 JSON

插件采集流程如下：

1. 用户手动打开小红书帖子详情页
2. 点击浏览器插件
3. 插件 popup 向当前页发送采集消息
4. `content.js` 从当前 DOM 提取帖子信息
5. popup 展示 JSON，并支持复制或下载
6. 用户回到分析台导入 JSON

自动填充流程如下：

1. 用户在分析台填写来源链接，或直接点击“根据当前页面自动填充”
2. 分析台通过 `window.postMessage` 向扩展桥接脚本发送请求
3. `page-bridge.js` 将请求转发给扩展后台 `background.js`
4. `background.js` 选择已打开的小红书标签页，或根据链接后台打开新标签页
5. 后台向目标小红书页面发送 `COLLECT_POST` 消息
6. `content.js` 提取帖子内容并返回
7. 扩展将结果回传给分析台页面
8. 分析台将标题、正文、作者、时间和链接自动填入录入表单

批量链接导入流程如下：

1. 用户在分析台粘贴多个帖子链接
2. 前端解析成去重后的链接列表
3. 前端按串行队列逐条调用现有扩展桥接
4. 每成功一条就立即写入帖子库并刷新分析结果
5. 失败链接记录到失败列表
6. 任务完成后展示汇总状态

## 6. 核心模块说明

### 6.1 分析核心

文件：`analysis-core.js`

作用：

- 提供统一的分析入口 `analyzePostRecords`
- 封装高频词统计、问题提取、流程观察和报告生成
- 供浏览器页面和 Node 测试共用

主要函数：

- `normalizeText(text)`：统一文本格式
- `countTopics(posts)`：统计高频关键词
- `extractQuestions(posts)`：提取常见面试题片段
- `extractFlows(posts)`：总结流程观察
- `generateReport(params)`：生成 Markdown 报告
- `analyzePostRecords(records, filters)`：分析入口

### 6.2 帖子结构与帖子库

文件：`app.js`

系统使用统一帖子结构：

```js
{
  id: "post-1713770000000-acde12",
  company: "腾讯",
  role: "后端开发实习",
  keyword: "腾讯 后端开发 面经",
  title: "腾讯后台开发实习一面面经",
  author: "小红薯A1",
  publishTime: "2026-04-20",
  sourceUrl: "https://www.xiaohongshu.com/...",
  content: "帖子正文",
  collectedAt: "2026-04-22T10:30:00.000Z"
}
```

关键函数：

- `createPostRecord(partial)`：补全标准帖子结构
- `loadPosts()`：从 `localStorage` 读取帖子库
- `persistPosts()`：持久化帖子库
- `mergePosts(existing, incoming)`：合并导入数据
- `addPostFromForm()`：将当前表单内容写入帖子库
- `importLibrary(file)`：导入 JSON
- `exportLibrary()`：导出 JSON
- `applyAutofillPayload(payload)`：把扩展返回的内容回填到录入表单
- `requestExtensionAutofillAsync(mode, sourceUrl)`：以 Promise 方式调用扩展桥接
- `startBatchImport()`：串行执行批量链接导入
- `stopBatchImport()`：停止批量任务
- `savePayloadToLibrary(payload)`：将自动填充结果直接写入帖子库

### 6.3 页面与扩展桥接

文件：`app.js`、`extension/page-bridge.js`

桥接方式：

- 分析台页面使用 `window.postMessage` 向桥接脚本发送 `OFFERSCOPE_AUTOFILL_REQUEST`
- 桥接脚本通过 `chrome.runtime.sendMessage` 转发给扩展后台
- 扩展后台返回后，桥接脚本再用 `window.postMessage` 把结果送回页面

说明：

- 该桥接只在 `http://localhost/*`、`http://127.0.0.1/*` 和 `http://[::1]/*` 上启用
- 如果直接双击打开本地文件，自动填充按钮不可用

### 6.4 浏览器插件采集

目录：`extension/`

关键文件：

- `manifest.json`：Manifest V3 配置
- `background.js`：统一处理自动填充请求
- `page-bridge.js`：分析台页面桥接
- `popup.html` / `popup.css`：插件界面
- `popup.js`：发起采集、复制 JSON、下载 JSON
- `content.js`：读取当前页面 DOM 并组装帖子记录

当前插件提取字段：

- 标题
- 正文
- 作者
- 发布时间
- 来源链接
- 标签
- 采集时间

说明：

- 插件只做“当前页采集”和“单链接自动填充”
- 批量链接导入由分析台串行调度多个“单链接自动填充”请求实现
- 不做自动翻页
- 不做批量抓取
- 不绕过登录或平台限制

## 7. 本地存储策略

帖子库使用浏览器 `localStorage` 持久化，键名为：

```text
offerscope-post-library-v1
```

优点：

- 不需要后端
- 刷新页面后数据仍保留
- 适合第一阶段验证

限制：

- 仅限当前浏览器本地环境
- 不适合大规模数据
- 不支持多端同步

## 8. 当前版本的限制

### 8.1 数据来源限制

当前版本不支持自动批量采集，只支持：

- 手工录入
- JSON 导入
- 浏览器插件采集当前页
- 分析台通过扩展按单链接自动填充
- 分析台通过扩展串行导入多个链接

### 8.2 分析精度限制

当前采用规则法而非 NLP 或大模型方法，因此存在以下问题：

- 同义表达不能自动归并
- 中文语义理解能力有限
- 去重能力较弱
- 关键词质量依赖主题词表和停用词表

### 8.3 插件稳定性限制

插件依赖小红书网页 DOM 结构，如果页面结构变化，采集选择器可能需要调整。

## 9. 测试方式

当前项目已加入一个基础测试文件：

```text
test-analysis.js
```

作用：

- 在 Node 环境直接调用 `analysis-core.js`
- 检查样本数、高频词、问题提取、流程观察和报告标题

建议测试分层：

- 分析核心：Node 脚本测试
- 前端页面：本地手工 smoke test
- 插件采集：浏览器手工验证
- 页面自动填充桥接：浏览器手工验证
- 批量链接导入：浏览器手工验证

## 10. 后续演进方向

建议按以下顺序继续推进：

1. 增加 `txt/csv` 文件导入
2. 增强帖子元信息结构
3. 增加本地历史分析记录
4. 接入大模型做相似问题聚类
5. 重构为前后端分离应用

## 11. 运行说明

### 11.1 分析台

直接打开 `index.html` 即可运行，或在项目目录执行：

```powershell
python -m http.server 8080
```

然后访问：

```text
http://localhost:8080
```

### 11.2 插件

1. 打开 `chrome://extensions/`
2. 开启“开发者模式”
3. 选择“加载已解压的扩展程序”
4. 选择项目中的 `extension` 目录

### 11.3 自动填充功能说明

- 如果只用手工录入，直接打开本地文件即可。
- 如果要使用分析台上的自动填充按钮，请通过 `http://localhost`、`http://127.0.0.1` 或 `http://[::1]` 方式打开页面。
- 自动填充属于“人工触发的单页采集”，不是批量抓取方案。

## 12. 开发建议

- 继续保持“采集”和“分析”分层，避免后续耦合
- 在接入大模型前，先定义稳定输出 schema
- 在增加更多导入方式前，先固定帖子数据结构
- 如果后续需要对比多个公司，建议增加筛选和命名分析快照功能
