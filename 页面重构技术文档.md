# OfferScope 页面重构技术文档

## 1. 目标

本轮重构解决两个核心问题：

1. 原页面把采集、帖子库、分析和大模型全部堆在同一屏，操作密度过高。
2. 新增“关键词搜索候选池 -> 手动挑选入库”后，原来的工具面板风格已经不适合内容型工作流。

因此当前版本的目标不是简单拆页，而是把项目调整为更贴近“小红书内容浏览 + 人工精选”的工作台：

- 用多路由拆分功能阶段
- 用本地服务承接 `XhsSkills-master` 的搜索与详情能力
- 用候选池把“抓到的帖子”和“确认入库的帖子”隔离开
- 用更轻、更白、更卡片化的界面承载内容筛选

## 2. 当前架构

### 2.1 前端

前端仍然保持静态页面架构，核心文件如下：

```text
demo_xiaohongshu/
├─ index.html
├─ styles.css
├─ router.js
├─ app.js
├─ analysis-core.js
├─ local_app_server.py
├─ 页面重构技术文档.md
├─ 页面重构计划.md
└─ 需求文档.md
```

职责划分：

- `index.html`
  - 提供应用壳、顶部操作区、导航区、全局概览区和内容挂载点
- `styles.css`
  - 提供整体视觉系统、响应式布局、卡片流、帖子候选卡和分析页样式
- `router.js`
  - 维护 hash 路由定义和路由解析
- `app.js`
  - 维护状态、页面渲染、事件绑定、本地存储、搜索请求、扩展桥接和大模型调用
- `analysis-core.js`
  - 维护规则分析逻辑，不关心 DOM

### 2.2 本地服务

由于 `XhsSkills-master` 的 `xhs-apis` 是 Python CLI，而当前项目是浏览器页面，因此新增本地桥接服务：

- 文件：[local_app_server.py](/d:/codex/demo_xiaohongshu/local_app_server.py:1)
- 主要职责：
  - 提供静态页面服务
  - 暴露搜索接口 `/api/xhs/search`
  - 暴露详情接口 `/api/xhs/note-detail`
  - 统一把 `xhs-apis` 返回的数据整理成前端更好消费的结构

## 3. 路由与页面分工

当前路由如下：

- `#/collect`
  - 关键词搜索、候选池挑选、手动录入、扩展自动填充、批量链接导入
- `#/library`
  - 本地帖子库查看、JSON 导入导出、删除、回填草稿
- `#/analysis`
  - 高频关键词、常见问题、流程观察、Markdown 报告
- `#/llm`
  - 大模型配置、问题回答、总结和最终报告

默认路由为 `#/collect`。

## 4. 数据流

### 4.1 搜索与入库

新的主流程为：

1. 用户在采集页输入关键词、Cookies、分页参数。
2. 前端调用 `/api/xhs/search`。
3. 本地服务调用 `XhsSkills-master/skills/xhs-apis/scripts/xhs_api_tool.py` 对应能力。
4. 服务端返回标准化候选结果。
5. 前端将结果渲染为卡片流。
6. 用户手动勾选或单条入库。
7. 入库后的数据写入浏览器本地帖子库。

### 4.2 分析与总结

1. 前端读取帖子库。
2. `analysis-core.js` 进行规则分析。
3. 分析页实时展示关键词、问题和流程观察。
4. 大模型页基于规则分析结果拼装 `chat/completions` 请求。
5. 模型返回 JSON 后，前端解析并回填答案与总结。
6. 最终 Markdown 报告由规则分析结果与大模型结果共同组成。

## 5. 状态设计

前端共享状态主要包括：

- `posts`
  - 已确认入库的帖子
- `xhsConfig`
  - Cookies、关键词、页码、页大小、排序方式
- `searchResults`
  - 当前候选池结果
- `selectedSearchResultIds`
  - 候选池勾选状态
- `llmConfig`
  - 大模型接口配置
- `llmResult`
  - 大模型回答和总结
- `batchJob`
  - 批量导入任务状态
- `pendingBridgeRequest`
  - 浏览器扩展自动填充请求状态

持久化策略：

- `posts`、`xhsConfig`、`llmConfig` 写入 `localStorage`
- `searchResults`、`selectedSearchResultIds`、`batchJob` 保持为会话内状态

## 6. UI 重构策略

这次 UI 调整不是后台管理台风格，而是“内容工作台”风格，具体策略如下：

### 6.1 视觉方向

- 主色使用更接近小红书气质的亮红色
- 背景更轻、更白，减少厚重玻璃感
- 卡片采用圆角大、边框轻、阴影浅的风格
- 操作按钮弱化为内容辅助，而不是视觉主角

### 6.2 信息层级

- 顶部只保留品牌、总操作和全局摘要
- 左侧固定流程导航和关键概览
- 采集页把候选池作为核心内容区
- 帖子候选结果改为更像内容卡片流的展示

### 6.3 与“小红书风格”的边界

当前实现是“小红书启发式风格”，不是对官方页面的照搬。重点是借用以下交互语言：

- 封面优先
- 卡片优先
- 内容优先
- 轻操作、强浏览

## 7. 兼容性与限制

### 7.1 运行要求

- 推荐使用本地 HTTP 服务打开项目
- 如果要使用关键词搜索，需要运行：

```powershell
cd d:\codex\demo_xiaohongshu
python local_app_server.py 8080
```

### 7.2 能力边界

- 搜索结果依赖有效的小红书 Cookies
- 扩展自动填充依赖浏览器扩展桥接
- 大模型调用依赖用户提供可用的兼容 `chat/completions` 接口

## 8. 测试策略

本轮改造后的验证重点：

1. 路由默认跳转与页面切换是否正常
2. 首页壳子是否正确渲染导航和挂载点
3. `app.js`、`router.js`、`analysis-core.js` 是否通过语法检查
4. `local_app_server.py` 是否能正常启动并返回 `/api/health`
5. 规则分析和报告生成是否保持可用

## 9. 后续建议

下一阶段建议继续拆分：

1. 把本地状态与持久化抽离为 `store.js`
2. 把大模型调用抽离为 `llm-service.js`
3. 把本地服务请求抽离为 `xhs-service.js`
4. 把各页面渲染拆成独立模块
5. 如果后续继续扩展交互复杂度，再考虑迁移到组件化框架

## 10. 2026-04-27 持久化与启动方式更新

本次实现已完成以下调整：

1. 数据持久化从浏览器 `localStorage` 迁移到本地 SQLite
2. 本地服务新增 `/api/storage/*` 接口作为统一数据入口
3. 前端启动时先请求 bootstrap 数据，再渲染依赖状态的页面
4. 当数据库为空时，前端会尝试把旧版浏览器数据一次性导入 SQLite
5. 新增桌面启动器 `desktop_launcher.py` 与 `start_offerscope.cmd`，用于免手动命令启动本地服务

当前新的持久化职责如下：
- `local_app_server.py`
  - 承载 XHS 搜索/详情接口
  - 承载 SQLite schema 初始化、元数据管理、帖子与配置持久化 API
- `store.js`
  - 改为异步 API 客户端
  - 仅在旧数据迁移阶段读取浏览器 `localStorage`
- `app.js`
  - 启动时执行 bootstrap
  - 所有长期数据写入统一走后端持久化接口

数据库默认文件位置：
- `data/offerscope.db`
