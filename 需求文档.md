# 小红书面经采集台需求文档

## 1. 项目背景

用户希望围绕“小红书面经内容”建立一条完整但轻量的工作流：

1. 输入关键词搜索面经帖子
2. 先查看候选结果，而不是直接自动入库
3. 由人工挑选值得保留的帖子样本
4. 再对已确认样本做问题提取、规则分析和大模型回答

项目当前不再只是“帖子录入工具”，而是一个面向内容筛选和求职复盘的本地分析台。

## 2. 当前阶段目标

当前阶段聚焦 MVP 闭环：

- 支持关键词搜索帖子
- 展示候选帖子详情
- 支持人工勾选入库
- 支持本地帖子库管理
- 支持规则分析
- 支持大模型回答与总结
- 用更接近小红书内容社区的风格承载整个流程

## 3. 核心用户场景

### 场景 1：关键词拉样本

用户输入“腾讯 后端开发 面经”等关键词，希望快速得到一批帖子候选结果，并先看内容质量，再决定是否纳入分析样本。

### 场景 2：人工精选入库

用户不希望抓到什么就存什么，而是想主动筛掉广告、重复、无效内容，只保留高质量帖子入库。

### 场景 3：面经整理与回答

用户在得到一批精选帖子后，希望系统自动提取常见问题，并用大模型给出更适合面试表达的回答和总结。

## 4. 功能需求

### 4.1 采集页

采集页需要提供：

- 公司、岗位、分析关键词配置
- 小红书 Cookies 输入
- 搜索关键词输入
- 页码、页大小、排序方式设置
- 发起搜索
- 展示候选帖子结果
- 勾选候选帖子
- 单条入库
- 批量勾选入库
- 手动录入帖子
- 从浏览器扩展读取当前页
- 根据链接自动填充
- 批量链接导入

### 4.2 候选池展示

候选帖子结果至少需要展示：

- 标题
- 作者
- 发布时间
- 点赞数、评论数、收藏数
- 封面图
- 摘要或正文片段
- 是否已勾选
- 加载详情按钮
- 单条入库按钮
- 打开原帖按钮

候选池中的数据默认不自动写入帖子库。

### 4.3 帖子库

帖子库需要支持：

- 查看所有已入库帖子
- 删除单条帖子
- 把某条帖子重新加载为草稿
- 导入 JSON
- 导出 JSON
- 清空帖子库

### 4.4 规则分析

分析页需要支持：

- 样本数量展示
- 高频关键词提取
- 高频问题提取
- 流程观察总结
- Markdown 报告预览

### 4.5 大模型能力

大模型页需要支持：

- 配置兼容 `chat/completions` 的接口地址
- 配置 API Key
- 配置模型名
- 配置系统提示词
- 把提取问题交给模型回答
- 生成整体总结
- 展示最终 Markdown 报告

## 5. 体验需求

### 5.1 页面风格

页面风格要从传统工具后台，调整为更像小红书内容浏览产品的体验：

- 轻背景
- 红色重点色
- 大圆角卡片
- 封面优先
- 内容优先
- 操作弱化
- 更适合浏览和挑选帖子

### 5.2 操作体验

- 用户进入首页后应优先看到关键词搜索与候选内容
- 候选池和已入库帖子要清晰区分
- 分析和总结流程要沿着导航自然推进
- 文案要保持中文清晰可读

## 6. 非功能需求

### 6.1 技术约束

- 当前仍采用静态前端 + 本地 Python 服务
- 不引入复杂后端系统
- 不引入用户账号体系
- 不做云端存储

### 6.2 可维护性

- 路由要独立维护
- 分析逻辑要与 UI 解耦
- 本地服务要与前端边界清晰
- 搜索结果结构要统一规范，便于后续扩展评论或作者页

## 7. 当前阶段验收标准

满足以下条件即可认为当前版本达标：

1. 首页能正常显示新的小红书风格壳子
2. 路由切换正常
3. 关键词搜索区可输入并发起请求
4. 搜索结果可展示为候选卡片
5. 可勾选并手动入库
6. 本地帖子库管理正常
7. 分析页和大模型页仍可正常使用
8. 本地服务和脚本检查通过

## 8. 后续扩展方向

下一阶段可以优先考虑：

1. 候选池增加全选、反选、条件过滤
2. 支持评论抓取并把评论问题一起纳入分析
3. 支持按作者或时间维度筛选样本
4. 增加搜索历史与分析快照
5. 增加从候选池直接预览完整详情的抽屉或弹层
