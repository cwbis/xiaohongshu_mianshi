## Context

当前帖子库页面只提供单一的平铺帖子列表，所有帖子按采集时间混排展示。虽然每条帖子已经保留了 `company`、`role`、`title`、`author`、`publishTime` 等字段，但这些信息目前只体现在单条卡片上，没有被组织成便于浏览的大纲结构。

这次变更聚焦在前端阅读和管理体验，而不是后端存储能力扩展。现有 SQLite、`/api/storage/bootstrap` 和帖子持久化接口已经能够返回分组所需字段，因此可以优先通过前端派生分组结果来完成能力建设，避免增加不必要的后端复杂度。

## Goals / Non-Goals

**Goals:**
- 让帖子库支持按公司聚合浏览，并在组内进一步区分岗位方向。
- 让用户可以在“分组概览”和“原始平铺列表”之间切换，兼顾管理和逐条阅读。
- 在分组视图中保留现有帖子管理动作，例如查看全文、载入草稿、删除。
- 为缺失 `company` 或 `role` 的帖子定义稳定的兜底分组标签，避免帖子丢失在视图之外。

**Non-Goals:**
- 不修改帖子数据库 schema，也不要求新增后端接口。
- 不引入复杂筛选器、全文搜索或标签系统。
- 不在这次变更中处理跨公司对比分析，只关注帖子库浏览与管理。
- 不要求持久化用户当前选择的视图模式或分组展开状态。

## Decisions

### 1. 采用“公司一级分组 + 岗位二级分组”的层级结构

帖子库分组视图先按 `company` 聚合，再在每个公司组内按 `role` 划分子组。每个子组展示帖子数量、最近采集时间和组内帖子列表。

选择原因：
- 用户最先关注的通常是目标公司，先按公司分组符合浏览心智。
- 同一公司往往对应多个岗位方向，岗位作为二级层级可以保持组内可读性。
- 这个结构可以直接从现有帖子字段派生，不需要额外维护分组元数据。

考虑过的替代方案：
- 只按公司分组：组内帖子过多时仍然杂乱。
- 只按“公司 + 岗位”拼接成单层分组：虽然实现简单，但概览层次较弱。
- 加入更多维度分组：当前阶段会显著增加 UI 复杂度。

### 2. 保留“分组视图 / 平铺视图”双模式

帖子库顶部增加视图切换控件。默认进入分组视图，同时允许用户切回原有平铺列表。

选择原因：
- 分组视图适合快速概览和按目标浏览。
- 平铺视图仍适合逐条扫描最新帖子，保留能减少使用断层。
- 双模式实现对现有逻辑侵入较小，可以复用已有帖子卡片结构。

考虑过的替代方案：
- 直接替换成分组视图：会让依赖原有顺序浏览的用户失去熟悉路径。
- 只提供分组视图不允许切换：灵活性不足。

### 3. 通过前端派生分组结果，不修改持久化模型

`renderLibraryPage` 相关逻辑在读取 `state.posts` 后，基于帖子数组即时生成分组数据结构，例如：
- `companyLabel`
- `roleLabel`
- `postCount`
- `latestCollectedAt`
- `posts[]`

选择原因：
- 现有 `state.posts` 已具备足够字段，不需要数据库新增列。
- 分组属于显示层组织逻辑，放在前端更容易迭代。
- 这样可以避免影响 `local-persistent-store` 既有行为与接口契约。

考虑过的替代方案：
- 后端直接返回分组结果：适合复杂聚合，但当前收益不高。
- 预计算分组并写回数据库：会让显示逻辑和存储耦合。

### 4. 定义缺失字段的兜底规则

当帖子缺少 `company` 或 `role` 时：
- `company` 为空时归入“未分类公司”
- `role` 为空时归入“未分类岗位”

选择原因：
- 保证所有帖子都能进入分组视图。
- 避免因数据质量不完整而出现“帖子消失”的误解。
- 用户可以通过组名快速识别哪些帖子需要后续补齐信息。

考虑过的替代方案：
- 跳过缺失字段帖子：会造成数据不可见。
- 把缺失帖子集中放入单独“其他”列表：信息过于模糊。

### 5. 组内管理动作复用现有帖子卡片行为

组内帖子仍然复用现有卡片管理动作，包括：
- 查看全文 / 收起全文
- 载入草稿
- 删除
- 打开原帖链接（如存在）

选择原因：
- 能保持行为一致，降低学习成本。
- 复用现有动作处理逻辑，减少回归风险。
- 让这次变更聚焦“组织与浏览”，而不是重写帖子管理能力。

## Risks / Trade-offs

- [分组视图会让页面层级变深] -> 通过保持组摘要简洁，并允许快速切换回平铺视图降低认知成本。
- [帖子很多时前端派生分组可能增加渲染负担] -> 先采用轻量数组聚合和按需展开，后续若数据量进一步增长再考虑虚拟化。
- [缺失字段兜底分组可能过大] -> 使用明确标签暴露数据质量问题，便于后续手动修正。
- [双视图模式会增加状态管理复杂度] -> 将视图模式和展开状态限制在帖子库页面局部状态中，不扩散到全局持久化。

## Migration Plan

1. 在帖子库页增加视图模式状态和分组数据派生方法。
2. 新增分组概览渲染与组内帖子渲染逻辑。
3. 将现有帖子管理动作接入分组视图。
4. 保留原平铺列表渲染作为备用模式。
5. 补充前端验证，确认删除、载入草稿、全文展开在两种视图下都工作正常。

回滚策略：
- 如果分组视图引入明显问题，可以保留代码但默认退回平铺视图。
- 如需快速回退，实现上只需移除帖子库视图切换入口并恢复原始渲染路径。

## Open Questions

- 默认视图是否应始终为分组视图，还是根据帖子数量动态选择？
- 组内帖子默认是否按 `collectedAt` 排序，还是允许切换为 `publishTime`？
- 是否需要在后续迭代中加入“只看某公司 / 某岗位”的快速筛选能力？
