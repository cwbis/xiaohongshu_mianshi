## Context

OfferScope 当前是一个以浏览器为主的本地工具。前端会通过 `store.js` 初始化 `posts`、`xhsConfig` 和 `llmConfig`，而这些数据目前直接读写浏览器 `localStorage`。与此同时，搜索和帖子详情能力已经依赖 `local_app_server.py`，但这个服务目前既不负责业务数据持久化，也需要用户在命令行中手动启动后才能进入完整工作流。

这次变更会同时影响前端状态层、后端服务层、本地运行时模型以及项目运维方式。我们需要一种比浏览器存储更稳定、迁移更容易、又足够轻量的持久化方式来支撑单机工作流；同时也需要一个新的启动路径，在保留现有 HTML/JS 前端的前提下，把服务启动细节从用户视角中隐藏掉。

## Goals / Non-Goals

**Goals:**
- 把需要长期保存的数据从浏览器 `localStorage` 迁移到应用托管的轻量数据库中。
- 为帖子和应用配置增加后端持久化 API，让前端只面对一个统一数据源。
- 在可行的情况下，把已有浏览器数据在首次运行时迁移到数据库中。
- 让用户在日常使用时不再需要手动执行 `python local_app_server.py`。
- 保持当前本地优先、单用户、小红书采集与分析工作流不变。

**Non-Goals:**
- 不引入多用户账号体系、远程同步或云端数据库基础设施。
- 不把前端整体重写成框架化项目，也不改动现有分析流程本身。
- 不在这次变更中处理完整签名安装包发布。
- 不替换现有 Python 桥接服务的技术栈。

## Decisions

### 1. 使用 SQLite 作为持久化层

后端将使用 Python 内置的 `sqlite3` 模块，并由 `local_app_server.py` 管理单个应用数据库文件。

选择原因：
- SQLite 足够轻量、内嵌，不需要额外部署独立数据库服务。
- Python 自带 `sqlite3`，部署成本低，新增依赖少。
- 它支持 schema 版本管理、唯一约束和事务写入，满足帖子去重与配置更新的需要。

考虑过的替代方案：
- JSON 文件：实现简单，但在 schema 演进、去重和并发读写上较弱。
- TinyDB 之类文件型库：可以用，但 SQLite 更标准，也能减少额外依赖。
- 远程 Postgres/MySQL：对于当前单机单用户工作流来说过重，运维成本也更高。

### 2. 引入显式持久化 API，并让前端改为异步加载

前端不再直接调用 `localStorage` 保存长期数据，而是通过新的本地 API 进行加载和保存，例如：
- `GET /api/storage/bootstrap`
- `GET /api/storage/posts`
- `PUT /api/storage/posts`
- `GET /api/storage/settings`
- `PUT /api/storage/settings/:scope`
- `POST /api/storage/import-local`

选择原因：
- 可以把校验、标准化、去重和迁移逻辑统一收敛到后端。
- 将来即使更换前端承载方式，持久化能力也不再依赖浏览器 API。
- 这与当前架构方向一致，因为前端本来就已经通过本地服务访问小红书数据能力。

考虑过的替代方案：
- 保留 `localStorage` 作为缓存，之后再同步：会形成双数据源，增加迁移复杂度。
- 让浏览器直接访问 SQLite/WASM 数据库：既不能解决服务启动问题，也会让文件访问更复杂。

### 3. 帖子和配置分表存储，并加入版本化迁移支持

数据库将包含以下结构：
- `posts`：保存标准化帖子记录，以应用级 `id` 为主键，并对 `source_url` 和/或 `note_id` 增加唯一约束（在字段可用时）。
- `settings`：按 scope 保存配置，例如 `xhsConfig` 和 `llmConfig`。
- `meta`：保存 schema 版本和迁移元数据。

选择原因：
- 帖子和配置的生命周期、更新方式不同，分开更清晰。
- `meta` 表有助于做受控、可测试的 schema 升级。
- 用 scope 保存配置可以避免为每种配置都单独扩表，灵活性更高。

考虑过的替代方案：
- 单表保存整个 JSON 大对象：虽然简单，但不利于查询、校验和局部更新。
- 每种配置单独建表：当前规模下过于刚性。

### 4. 只做一次浏览器数据迁移，迁移后以 SQLite 为唯一真源

在启动时，如果后端发现数据库为空，而浏览器中仍存在旧版数据，则前端/运行时会触发一次性导入。导入成功后，后续的读取和写入都只通过后端 API 完成。

选择原因：
- 可以尽量保留现有用户数据，避免要求用户手动导出再导入。
- 可以避免长期双写带来的复杂性，把数据源尽快收敛为一个。

考虑过的替代方案：
- 只支持手动导出/导入：实现更保守，但用户体验较差。
- 长期保持浏览器和数据库双向同步：没有必要，且容易产生数据漂移。

### 5. 使用轻量 Python 桌面壳自动启动本地服务

推荐运行路径是增加一个基于 `pywebview` 的轻量 Python 桌面启动器。启动器将：
- 以内嵌线程或后台方式启动 `local_app_server.py`
- 等待 `/api/health` 就绪
- 以本地原生窗口打开指向本地服务的 UI
- 在窗口关闭时停止本地服务

选择原因：
- 可以最大化复用现有 Python 后端和静态前端，重构成本低。
- 用户无需打开终端手动运行服务。
- 相比 Electron 更轻，也不需要像 Tauri 那样引入新的工具链。

考虑过的替代方案：
- 纯 `.bat`/快捷方式启动器：比手动命令好一些，但仍然暴露命令行式体验。
- Electron/Tauri 外壳：能力足够，但对当前阶段来说偏重。
- OS 级后台服务/守护进程：后续可以考虑，但当前跨平台和运维复杂度更高。

## Risks / Trade-offs

- [SQLite 仍然会把数据保存在本机磁盘，而不是浏览器中] -> 需要在文档里明确这一假设，同时保持持久化 API 设计可扩展，便于未来替换成远程存储实现。
- [前端启动流程变成异步] -> 通过增加加载态和单一 bootstrap 接口，避免页面在状态未就绪时局部渲染。
- [一次性迁移可能遗漏格式异常的旧数据] -> 在后端做标准化与校验，跳过非法记录并记录日志，同时保留手动 JSON 导入作为兜底。
- [桌面壳会新增一个 Python 依赖] -> 选择最小依赖 `pywebview`，并保留浏览器访问模式作为开发兜底路径。
- [数据库文件损坏或 schema 缺陷会影响全部数据] -> 通过事务写入、schema 版本控制和后端 JSON 导出能力降低恢复成本。

## Migration Plan

1. 在本地服务中加入 SQLite schema 初始化和 repository 帮助层。
2. 实现持久化 API，并补齐标准化、去重和 settings scope 读写逻辑。
3. 把 `store.js` 重构为 API 驱动客户端，同时更新 `app.js` 的启动和保存流程。
4. 增加从浏览器 `localStorage` 向 SQLite 的一次性迁移逻辑。
5. 增加桌面启动器，在打开应用时自动拉起本地服务和 UI。
6. 保留 `python local_app_server.py` 作为开发和回滚场景的兜底启动方式。

回滚策略：
- 如果桌面壳路径失败，用户仍可以退回到现有浏览器 + 本地服务的工作流。
- 如果数据库持久化引入问题，可以依赖 JSON 导出能力和保留的旧启动路径进行恢复，并在开发阶段临时回退到旧版读取方式。
